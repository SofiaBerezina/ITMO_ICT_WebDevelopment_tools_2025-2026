Лабораторная работа 2: Потоки. Процессы. Асинхронность

Цель работы

Понять разницу между потоками, процессами и асинхронностью в Python. Научиться применять каждый подход на практике.

Задание 1. Сумма чисел от 1 до 10 триллионов

Чтобы посчитать сумму всех целых чисел от 1 до 10 000 000 000 000, программа разбивает эту задачу на 8 частей и считает параллельно.

threading — версия с потоками

Программа создаёт 8 потоков, каждый поток считает сумму своего диапазона чисел. Потоки используют общую память и работают внутри одного процесса. Из-за GIL в Python одновременно может исполняться только один поток, поэтому параллельность на самом деле не достигается. За счёт того, что задача решается по формуле арифметической прогрессии, каждый поток выполняется очень быстро.

Результат: 0.0005 секунд

multiprocessing — версия с процессами

Программа создаёт 8 процессов, каждый процесс живёт в своём адресном пространстве. Процессы не мешают друг другу и могут выполняться на разных ядрах процессора. Результаты вычислений передаются через очередь.

Результат: 0.0896 секунд

asyncio — асинхронная версия

Асинхронная программа создаёт 8 задач, которые выполняются в одном потоке. Для вычислительных задач асинхронность не даёт преимущества. За счёт того, что вычисления идут по формуле, они выполняются практически мгновенно.

Результат: 0.0003 секунд

Результаты задания 1

Подход Время (секунд)
threading 0.0005
multiprocessing 0.0896
asyncio 0.0003

Почему multiprocessing медленнее?

Все три программы для ускорения работы используют формулу суммы арифметической прогрессии: (первое_число + последнее_число) * количество_чисел // 2.

Из-за этого каждый поток или процесс завершает свою работу почти мгновенно. Время выполнения начинает зависеть не от вычислений, а от накладных расходов на создание процессов и потоков.

  • threading и asyncio работают в одном процессе, поэтому создаются быстро, и 0.0003–0.0005 секунд — это в основном время на разбиение задачи и вывод результатов.
  • multiprocessing тратит время на создание 8 отдельных процессов и передачу данных через очередь. Это гораздо более тяжёлая операция, поэтому результат — 0.0896 секунд.

Чтобы увидеть реальную разницу, нужно было бы считать сумму через цикл, а не через формулу. Но для демонстрации механизма работы достаточно и того, что multiprocessing создаёт процессы медленнее, чем threading создаёт потоки.

Вывод по заданию 1

Разница во времени выполнения между потоками и процессами в данном случае объясняется не реальными вычислениями, а скоростью создания дочерних объектов — потоков и процессов.

Задание 2. Параллельный парсинг веб-страниц

База данных из лабораторной работы 1

В первой лабораторной работе была создана база данных для тайм-менеджера. В ней есть следующие таблицы:

  • user — пользователи системы (логин, почта, хэш пароля)
  • priority — приоритеты задач (низкий, средний, высокий)
  • status — статусы задач (ожидает, выполняется, завершена)
  • tasktype — типы задач (работа, учёба, личное)
  • task — сами задачи (название, описание, дедлайн, затраченное время)
  • timelog — логи времени, потраченного на задачу
  • notification — уведомления о дедлайнах

Для второй лабораторной работы мы будем заполнять таблицу task. При парсинге веб-страницы мы извлекаем её заголовок и сохраняем в базу как новую задачу. Тип задачи — «работа» (work), статус — «ожидает» (pending), приоритет — «средний» (medium). Пользователь, создающий задачу, имеет id = 1.

Подключение к базе данных

В файле connection.py настраивается подключение. В первой лабораторной работе использовался PostgreSQL. Для парсинга нам проще использовать SQLite, чтобы не поднимать отдельный сервер. SQLite хранит базу данных прямо в файле, что удобно для тестирования.

# connection.py
from sqlmodel import SQLModel, Session, create_engine
import os

db_url = "sqlite:///./time_manager.db"
engine = create_engine(db_url, echo=True)

def init_db():
    SQLModel.metadata.create_all(engine)

def get_session():
    with Session(engine) as session:
        yield session

Модели данных импортируются из файла models.py, созданного в первой лабораторной работе. Они описывают все таблицы: User, Task, Priority, Status, TaskType, TimeLog, Notification.

Перед запуском парсинга вызывается init_db(), которая создаёт все таблицы, если их ещё нет. После этого можно сохранять спарсенные данные.

Программа парсит три сайта: python.org, github.com и stackoverflow.com. С каждого сайта извлекается заголовок страницы, который сохраняется в базу данных как новая задача. База данных используется из первой лабораторной работы.

Подход 1: threading (потоки)

Потоки хорошо подходят для парсинга, потому что основной задачей является ожидание ответа от сервера. Во время ожидания поток отдает управление другим потокам.

Создается отдельный поток для каждого URL. В потоке отправляется HTTP-запрос, парсится HTML и результат сохраняется в базу данных.

Подход 2: multiprocessing (процессы)

Процессы тоже справляются с парсингом, но они тяжелее потоков. Для каждого URL создается новый процесс. Это требует больше памяти и времени на создание, чем потоки.

Подход 3: asyncio (асинхронность)

Асинхронность — лучший выбор для парсинга. Все запросы отправляются одновременно, и программа не ждет ответа от каждого сервера по очереди. Операция сохранения в базу данных выносится в отдельный поток, чтобы не блокировать основной цикл событий.

Результаты выполнения

Подход Время (сек)
threading 0.92 сек
multiprocessing 1.11 сек
asyncio 0.61 сек

Анализ результатов

asyncio оказался самым быстрым. Он отправляет все запросы сразу и обрабатывает ответы по мере их поступления. Нет накладных расходов на создание потоков или процессов.

threading занял второе место. Создание потоков быстрее, чем создание процессов, но потоки все равно немного тяжелее, чем асинхронные задачи.

multiprocessing показал худший результат. Создание процессов требует больше времени и памяти, что для простых сетевых запросов неоправданно.

Общие выводы

  1. Для вычислительных задач лучше всего подходит multiprocessing. Процессы обходят GIL и реально используют все ядра процессора.

  2. Для задач с сетевыми запросами лучший выбор — asyncio. Он легкий и эффективный, не создает лишних потоков и процессов.

  3. threading занимает промежуточное положение. Он подходит для простых I/O задач, если не хочется усложнять код асинхронностью, но будет медленнее asyncio.

  4. GIL — главное ограничение Python. Из-за него потоки не ускоряют вычисления. Для тяжелой математики нужно использовать процессы.

  5. При выборе подхода нужно смотреть на задачу:

  6. Много сетевых запросов => asyncio
  7. Тяжелые вычисления => multiprocessing
  8. Простая I/O задача => threading
